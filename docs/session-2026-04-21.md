### 2. Babylon renderer optimisation

#### The problem
`SpriteRendererBabylon.ts` was creating one `MeshBuilder.CreateBox` per solid voxel. For a 7×7×7 sprite that's up to 343 separate Babylon `Mesh` objects, each with its own draw call. It was also creating a separate `StandardMaterial` per palette entry.

#### The solution
Replaced the whole approach with a **single merged mesh** built from a `VertexData` object, with two key optimisations:

**Face culling** — For each solid voxel, only emit a face if the neighbouring voxel in that direction is empty. Interior faces shared between two solid voxels are never generated. This typically cuts triangle count by 50–70%.

**Vertex colors** — Instead of one Babylon `StandardMaterial` per palette entry, the palette color is baked directly into each vertex as an RGBA value. The whole sprite uses a single `StandardMaterial` with `diffuseColor = White`, so all geometry renders in **one draw call** regardless of how many colors are used.

## The three files

### `src/engine/renderers/babylon/generateVoxelMesh.ts`
Pure function — takes a `Frame`, `Dimensions`, and `Palette`, returns a Babylon `VertexData`.

Key details:
- Voxel index formula: `x + y * dims.x + z * dims.x * dims.y` (matches the engine's `getVoxelIndex` util)
- World position: `wx = xi - centerX`, `wy = yi - centerY`, `wz = -(zi - centerZ)` — note the **Z flip**
- Because of the Z flip, the world +Z face corresponds to the voxel `zi-1` neighbour, and world -Z to `zi+1`
- 6 face definitions as `const` data (offsets, outward normal, neighbour delta in voxel space)
- Winding order follows Babylon.js left-hand convention, matching what `MeshBuilder.CreateBox` generates

### `src/engine/renderers/babylon/createSpriteMesh.ts`
Two small helpers:
- `createSpriteMesh(scene, vertexData, parent)` — creates a `Mesh`, parents it, applies vertex data
- `createSpriteMaterial(scene)` — creates the single shared `StandardMaterial` with `diffuseColor = White` and `backFaceCulling = false` (see known issue below)

### `src/engine/renderers/SpriteRendererBabylon.ts`
Rewritten to use the above. Key changes from the old version:
- `cubes: Mesh[]` and `materials: Map<number, StandardMaterial>` are gone
- Now holds `spriteMesh: Mesh | null` and `spriteMaterial: StandardMaterial`
- `updateMesh(frame?)` disposes the old mesh and creates a new one via `generateVoxelMesh`
- `updateMaterials(palette?)` updates `spriteData.palette` then calls `updateMesh` — colors are baked so the mesh must be rebuilt when the palette changes
- `currentFrame` is tracked so `updateMaterials` knows which frame to rebuild

## Known issue — face winding

`backFaceCulling = false` is set on the material as a temporary fix. Some faces have incorrect winding (vertices in the wrong order), which causes Babylon's backface culling to hide them. With culling disabled all faces are visible, but:
- Lighting is slightly wrong on back-facing triangles (normals are still outward, so it's subtle)
- Very minor extra GPU cost (drawing both sides of each face)

**To fix properly:** compare each of the 6 face definitions in `generateVoxelMesh.ts` against what `MeshBuilder.CreateBox` generates and correct any that differ. Once all 6 are correct, remove `backFaceCulling = false` from `createSpriteMaterial`.
